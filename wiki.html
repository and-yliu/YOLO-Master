YOLO-Master MoE Modules: Deep Dive & Evolution

This document provides an in-depth analysis of the Mixture-of-Experts (MoE) modules in ultralytics/nn/modules/moe.py.

Core MoE Modules Overview

ES_MOE (Early Stage / Efficient Sparse MoE)

The earliest MoE implementation, primarily for proof of concept.

Features:

  • Heterogeneous Experts: Different kernel sizes (3×3, 5×5, 7×7) as experts
  • Basic Routing: DynamicRoutingLayer based on global average pooling
  • Training Mode: Dense Forward (computes all experts and weights them)
  • No Shared Expert: Lacks shared path, training unstable early on

Pros: Intuitive structure
Cons: Slow training, high memory, prone to routing collapse

OptimizedMOE

First major improvement over ES_MOE.

Features:

  • Shared Expert: Always-active parallel branch — modern MoE standard
  • Efficient Spatial Routing: EfficientSpatialRouter with pre-pooling
  • Homogeneous Experts: Same SimpleExpert for parallel optimization

Pros: More stable, higher efficiency
Cons: Low configurability

OptimizedMOEImproved (ModularRouterExpertMoE)

Currently Recommended Version. Highly modular, industrial-grade.

Features:

  • Highly Modular: Pluggable Router (Efficient/Local/Adaptive) and Expert (Simple/Ghost/Inverted)
  • Z-Loss: Prevents router logit explosion
  • Standardized Aux Loss: Load Balancing + Z-Loss
  • Init Strategy: Gaussian std=0.01 prevents "winner-takes-all"

Pros: High flexibility, good stability, production-ready
Cons: Inference speed can be improved

UltraOptimizedMoE

Designed for high-performance deployment.

Features:

  • UltraEfficientRouter: Depthwise Separable Conv + aggressive downsampling (8×/16×), routing FLOPs ↓ ~95%
  • Batched Expert Computation: Eliminates Python for loops, inference 3-5× faster
  • GroupNorm: Replaces BatchNorm, stable for micro-batch
  • Conditional Computation: Weight threshold skips low-contribution experts

Pros: Extremely fast, lowest FLOPs, minimal memory
Cons: Highest complexity

Router Modules Comparison

Module Features Scenario
DynamicRoutingLayer Basic, Soft/Hard Top-K ES_MOE only, not recommended
EfficientSpatialRouter Downsample then route General, balance accuracy/speed
LocalRoutingLayer Small downsampling ratio Small object detection
AdaptiveRoutingLayer Adaptive pool to 1×1 Extreme resource constraints
UltraEfficientRouter DSConv + aggressive downsample Edge device, extreme speed

Expert Modules Comparison

Module Structure Pros Cons
SimpleExpert Conv-BN-SiLU-Conv-BN Standard, easy to optimize Standard params
GhostExpert GhostNet-style cheap ops Params & FLOPs halved Slightly weaker features
InvertedResidualExpert MobileNetV2 inverted residual Mobile-friendly Deeper, more latency
OptimizedSimpleExpert GroupNorm instead of BN Small-batch stable Cannot fuse into Conv

Technical Evolution

  1. Heterogeneous → Homogeneous: Better hardware parallelism
  2. Stability First: Shared Expert solves training instability
  3. Compute Efficiency: Dense Forward → Sparse Forward → Batch Parallelism
  4. Numerical Stability: Z-Loss + specific initialization

Hardware Deployment

NVIDIA GPU (Server/Desktop)

  • Recommended: UltraOptimizedMoE
  • Batched Computation aggregates samples per expert, launches CUDA kernels at once
  • TensorRT 8.x+ supports TopK/Gather/Scatter well
  • Export: opset_version=13+

CPU (Intel/AMD)

  • Ideal for MoE — strong branch prediction
  • Sparse means skipping 50% FLOPs
  • Use OpenVINO or ONNX Runtime with use_sparse_inference=True

Mobile & Edge (NPU/DSP)

  • Challenge: Dynamic control flow weak on NPU
  • Solution A: GhostExpert halves memory bandwidth
  • Solution B: Static export — set top_k = num_experts
  • Solution C: Shared Expert Only for ultra-low power

Export Tips

  1. opset_version=13 or higher
  2. Fix Batch Size for best performance
  3. Verify atol=1e-4 after export

Configuration & FAQ

YAML Example:

YAML 
# Standard (recommended)
- [-1, 1, ModularRouterExpertMoE, [512, 4, 2]]
# Speed oriented (Ghost)
- [-1, 1, UltraOptimizedMoE, [512, 4, 2, 'ghost']]
# Extreme precision (8 experts)
- [-1, 1, ModularRouterExpertMoE, [512, 8, 2]]

Q1: Loss oscillation or NaN early?
→ Use OptimizedMOEImproved+ with Shared Expert & Z-Loss. Lower LR or increase warmup.

Q2: ONNX export "Gather/Scatter not supported"?
→ Set opset_version=13+. For old engines, use static export.

Q3: Higher VRAM than normal YOLO?
→ Parameters increase with N experts. Reduce num_experts or use GhostExpert. Use AMP.
